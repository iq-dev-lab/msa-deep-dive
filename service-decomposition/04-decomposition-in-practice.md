# 04. 서비스 분해 실전 — 전자상거래 도메인 예시

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 전자상거래 도메인을 어떤 기준으로 서비스로 나누는가?
- 주문, 결제, 재고, 배송, 회원, 상품을 왜 그 경계에서 분리하는가?
- 서비스 경계를 처음 잘못 그었을 때 어떻게 재조정하는가?
- "분리 후 합치기"와 "합친 것 다시 나누기"는 언제 발생하는가?
- 서비스 경계 설정 실수를 조기에 발견하는 신호는 무엇인가?

---

## 🔍 왜 이 개념이 MSA에서 중요한가

이론적 원칙(Bounded Context, SRP)을 실제 도메인에 적용하는 것은 별도의 기술이다. 전자상거래 도메인은 MSA 예시로 가장 많이 쓰이지만, 실제 경계 설정은 생각보다 복잡하다. 주문과 결제를 왜 분리하고, 재고와 배송을 왜 따로 두는지, 그 근거를 설명할 수 있어야 다른 도메인에서도 경계를 올바르게 그을 수 있다.

---

## 😱 흔한 실수 (Before — 직관적으로 나눌 때)

```
직관적 분해의 실수:

"주문할 때 결제도 하니까 주문+결제 묶기"
  → 결제 정책 변경(PG사 교체) 시 주문 서비스 재배포 필요
  → 결제 팀과 주문 팀이 같은 코드베이스에서 충돌

"재고는 상품 정보랑 관련 있으니 상품 서비스에 포함"
  → 상품 팀(마케팅/MD)과 재고 팀(물류/운영)이 다른 팀
  → 재고 로직 변경이 상품 검색 기능에 영향

"배송과 주문은 묶어야 주문 상태를 추적할 수 있음"
  → 배송사 API 변경 시 주문 로직 수정 필요
  → 주문 취소 시 배송 취소 트랜잭션 복잡도 폭발
```

## 🔬 내부 동작 원리 — 각 서비스 경계의 근거

### 1. 주문 서비스 — 주문의 생명주기

```
담당: 주문 생성, 조회, 취소, 상태 관리

경계의 근거:
  주문은 독립적인 생명주기를 가진다
  PENDING → PAYMENT_PENDING → CONFIRMED → SHIPPING → DELIVERED → CANCELLED

변경 이유: 주문 유효성 검증 규칙, 주문 상태 흐름, 이력 보관 정책
변경하지 않는 것: 결제 방식(결제 서비스), 재고 수량(재고 서비스), 배송 추적(배송 서비스)
```

```sql
-- order_db
CREATE TABLE orders (
    order_id    VARCHAR(36) PRIMARY KEY,
    customer_id VARCHAR(36) NOT NULL,      -- 참조만 (물리 FK 없음)
    status      ENUM('PENDING', 'CONFIRMED', 'CANCELLED', ...),
    total_amount DECIMAL(10,2),
    created_at  TIMESTAMP
);

CREATE TABLE order_items (
    item_id       VARCHAR(36) PRIMARY KEY,
    order_id      VARCHAR(36),
    product_id    VARCHAR(36) NOT NULL,     -- 참조만 (물리 FK 없음)
    product_name  VARCHAR(100) NOT NULL,    -- 주문 당시 스냅샷
    ordered_price DECIMAL(10,2) NOT NULL   -- 주문 당시 가격 스냅샷
);
```

### 2. 결제 서비스 — 금융 도메인 격리

```
담당: 결제 처리, 환불, 결제 수단 관리, 결제 내역

경계의 근거:
  결제는 금융 도메인 → PCI DSS 보안 규정 적용
  PG사 교체(KG이니시스 → 토스페이먼츠) 시 결제 서비스만 수정
  카드 정보는 결제 서비스 DB에만 저장

Saga 흐름:
  OrderPlaced 이벤트 수신
  → 결제 처리 시도
  → PaymentCompleted 이벤트 발행 (성공)
  또는 PaymentFailed 이벤트 발행 (실패 → 주문 취소 보상)
```

### 3. 재고 서비스 — 동시성 제어 집중

```
담당: 재고 조회, 차감, 복원, 알림

경계의 근거:
  재고는 물류 도메인 (담당 팀: 물류/운영팀)
  동시에 100명이 마지막 재고 1개를 주문할 때:
  → 재고 서비스 내부에서 SELECT FOR UPDATE 또는 Optimistic Lock
  → 다른 서비스가 재고를 직접 수정하면 이 동시성 제어가 깨짐
```

### 4. 상품 카탈로그 — 검색 특화 DB

```
담당: 상품 등록/수정/삭제, 검색, 카테고리 관리

재고와 분리하는 이유:
  상품 정보 변경 이유: 상품명, 이미지 수정 (MD팀)
  재고 변경 이유: 입고, 주문 차감, 재고 실사 (운영팀)
  → 다른 팀, 다른 이유 → 다른 서비스

DB 기술:
  기본 정보: MySQL (상품 ID, 가격)
  검색: Elasticsearch (풀텍스트 검색, 필터링)
  → Polyglot Persistence (Chapter 3 연결)
```

## 💻 서비스 경계 재조정 — 실제 사례

### 사례 1: 분리 후 합치기

```
초기 설계: 쿠폰 서비스 + 포인트 서비스 (별도)

문제 발생:
  주문 결제 시 쿠폰과 포인트 동시 적용 필요
  → 두 서비스에 동시 업데이트 → 분산 트랜잭션 발생
  → 항상 같이 배포됨 (배포 결합)

재조정:
  쿠폰 + 포인트 → 혜택 서비스(Benefit Service)로 통합
  → 하나의 트랜잭션으로 처리, 배포 단순화

신호: "두 서비스가 항상 같이 배포된다" → 합칠 후보
```

### 사례 2: 합친 것 다시 나누기

```
초기 설계: 주문+결제 통합 서비스

문제 발생:
  주문팀 3명 + 결제팀 3명이 같은 코드베이스
  결제 PG사 교체 시 주문 기능 전체 테스트 실행 필요
  코드 충돌 주 2회 이상

재조정:
  주문 서비스 / 결제 서비스 분리
  → 주문: OrderPlaced 이벤트 발행
  → 결제: 이벤트 구독하여 처리

신호: "팀이 다른데 같은 서비스를 수정한다" → 나눌 후보
```

### 경계 재조정 조기 발견 신호

```
⚠️ 재조정이 필요한 신호:

1. 기능 구현에 서비스 2개 이상 항상 수정
2. 서비스 A의 테스트가 서비스 B의 동작에 의존
3. 배포 순서가 고정됨
4. 데이터 조합을 위해 N+1 서비스 호출
```

## 📊 최종 전자상거래 서비스 맵

| 서비스 | 담당 팀 | DB 기술 | 변경 이유 |
|--------|---------|---------|----------|
| 주문 | 주문팀 | MySQL | 주문 비즈니스 로직 |
| 결제 | 결제팀 | MySQL | PG사 변경, 결제 수단 추가 |
| 재고 | 물류팀 | MySQL | 재고 정책, 동시성 처리 |
| 배송 | 배송팀 | MySQL | 배송사 연동 변경 |
| 상품 카탈로그 | 상품팀(MD) | MySQL + ES | 상품 정보, 검색 고도화 |
| 회원 | 플랫폼팀 | MySQL | 인증/인가 정책 |

## ⚖️ 트레이드오프

```
경계를 처음부터 완벽하게 그을 수 없다:
  도메인 지식이 쌓일수록 더 좋은 경계가 보임
  → 초기에는 약간 크게 설계하고, 문제가 보이면 나누기

너무 이른 분해의 비용:
  잘못된 경계에서 분산 트랜잭션, 과도한 API 호출 발생
  → 재조정 비용 (데이터 마이그레이션, API 재설계)

권장 전략:
  "모듈러 모놀리스로 시작 → 경계가 명확해지면 서비스로 추출"
```

## 📌 핵심 정리

```
✅ 전자상거래 6개 핵심 서비스
   주문 / 결제 / 재고 / 배송 / 상품카탈로그 / 회원

✅ 각 경계의 근거
   - 주문: 주문 생명주기 + 주문팀 소유
   - 결제: 금융 도메인, PG사 독립 교체
   - 재고: 물류 도메인, 동시성 제어 집중
   - 배송: 외부 물류사 연동 격리
   - 상품: MD팀 소유, 검색 특화 DB

✅ 경계 재조정 신호
   - 분리 후 합치기: 항상 같이 배포되는 서비스
   - 합친 것 나누기: 다른 팀이 같은 서비스 수정
```

## 🤔 생각해볼 문제

**Q1.** 쿠폰 서비스와 포인트 서비스를 별도로 운영 중인데, 주문 결제 시 동시에 적용해야 한다. 어떤 방식으로 설계하겠는가?

<details>
<summary>해설 보기</summary>

두 가지 접근이 있다. 첫째, 혜택 서비스로 통합: 쿠폰과 포인트가 항상 같은 트랜잭션에서 처리된다면 경계가 잘못 그어진 것이다. 혜택 서비스로 통합하면 하나의 트랜잭션으로 처리 가능하다. 둘째, Saga로 처리: 쿠폰과 포인트 변경 이유가 확실히 다르다면 Saga Choreography로 처리한다. 어느 방식이 옳은지는 팀 구조와 변경 이유를 따져봐야 한다.

</details>

**Q2.** 상품 이름이 변경됐을 때, 기존 주문의 상품명이 바뀌어서는 안 된다. 어떻게 설계하는가?

<details>
<summary>해설 보기</summary>

주문 시점의 상품명을 `order_items` 테이블의 `product_name` 컬럼에 스냅샷으로 저장한다. 이후 상품 정보가 바뀌어도 주문 이력은 변하지 않는다. `product_id`는 참조 목적으로 유지하되, 물리적 외래키는 사용하지 않는다.

</details>

**Q3.** 관리자 페이지(Admin)를 별도 서비스로 분리해야 하는가?

<details>
<summary>해설 보기</summary>

관리자 기능은 서비스가 아니라 각 서비스의 관리자용 엔드포인트로 두는 것이 일반적이다. 별도 Admin 서비스를 만들면 각 도메인의 데이터에 직접 접근해야 하는데, 이는 Shared Database 안티패턴이나 과도한 API 호출로 이어진다. 여러 서비스의 데이터를 조합하는 대시보드는 읽기 전용 BFF 레이어로 구성한다.

</details>
