# 03. 분산 모놀리스 안티패턴 — 쪼갰지만 여전히 결합된 이유

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 분산 모놀리스(Distributed Monolith)란 무엇이며, 왜 모놀리스보다 나쁜가?
- Shared Database가 서비스 독립성을 어떻게 파괴하는가?
- 동기 호출 체인이 가용성을 수학적으로 얼마나 낮추는가?
- 분산 모놀리스인지 아닌지 판단하는 기준은 무엇인가?
- 이미 분산 모놀리스가 된 시스템을 어떻게 탈출하는가?

---

## 🔍 왜 이 개념이 MSA에서 중요한가

많은 팀이 "MSA로 전환했다"고 생각하지만 실제로는 분산 모놀리스를 만들었다. 서비스를 물리적으로 분리했지만 데이터와 배포는 여전히 결합돼 있다. 분산 모놀리스는 모놀리스의 단점(배포 결합, 팀 결합)을 그대로 유지하면서 MSA의 단점(운영 복잡도, 네트워크 레이턴시)까지 추가한, 두 아키텍처의 최악을 합친 결과물이다.

---

## 😱 흔한 실수 (Before — 분산 모놀리스가 만들어지는 과정)

```
팀 회고: "MSA로 전환했는데 왜 배포가 더 불편해졌지?"

전환 전:
  하나의 Spring Boot 앱 → 한 번 배포

전환 후 (분산 모놀리스):
  주문 서비스, 결제 서비스, 회원 서비스 → 3개 서비스

  그런데:
  - 세 서비스가 같은 MySQL DB를 공유
  - 주문 테이블 스키마 변경 → 결제, 회원 코드도 수정 필요
  - 결제 서비스 배포 전에 주문 서비스 먼저 배포해야 하는 순서 의존성
  - 결국 3개 서비스를 동시에 배포해야 안전함

결과: 배포 단위만 3개로 늘었고 조율 비용 3배, 운영 복잡도 3배
```

---

## ✨ 올바른 접근 (After — 진짜 독립 서비스)

```
진짜 독립 서비스의 기준:

1. 데이터 독립: 각 서비스가 자신만의 DB를 소유
   → 주문 서비스 스키마 변경이 결제 서비스에 영향 없음

2. 배포 독립: 언제든 단독으로 배포 가능
   → 결제 서비스 배포가 주문 서비스 재시작을 요구하지 않음

3. 런타임 독립: 다른 서비스가 다운돼도 자신은 동작
   → 결제 서비스 장애 시 상품 조회는 정상 동작

4. 장애 격리: 한 서비스의 버그가 다른 서비스로 전파 안 됨
   → Circuit Breaker로 장애 전파 차단
```

---

## 🔬 내부 동작 원리 — 3가지 핵심 안티패턴

### 안티패턴 1: Shared Database (공유 데이터베이스)

```
분산 모놀리스 — Shared Database:

주문 서비스 ──┐
결제 서비스 ──┼──→ 공유 MySQL (같은 DB, 같은 스키마)
회원 서비스 ──┘

문제 시나리오:
  주문팀: "orders 테이블에 delivery_address 컬럼 추가해도 돼?"

  → 결제팀 확인 필요: "우리도 orders 테이블 쓰는데..."
  → 회원팀 확인 필요: "join 쿼리에서 orders 참조해서..."
  → 배포 순서 조율 필요: "먼저 orders 스키마 마이그레이션 후
                           모든 서비스 동시 재시작"

결과:
  스키마 변경 = 전체 팀 미팅 + 조율 + 동시 배포
  → 모놀리스와 완전히 동일한 배포 결합
  → 게다가 네트워크 레이턴시까지 추가됨
```

```sql
-- 결제 서비스가 직접 orders 테이블을 JOIN (안티패턴)
SELECT o.order_id, o.customer_id, o.total_amount, p.payment_status
FROM orders o
JOIN payments p ON o.order_id = p.order_id
WHERE o.status = 'PENDING';

-- 이 쿼리가 있는 한:
-- 1. orders 테이블 컬럼명 변경 → 결제 서비스 쿼리 깨짐
-- 2. orders 테이블 인덱스 변경 → 결제 서비스 성능에 영향
-- 3. 주문 DB 서버 다운 → 결제 서비스도 다운
```

### 안티패턴 2: 동기 호출 체인 — 가용성의 수학적 붕괴

**가용성 계산:**

```
서비스 하나의 가용성: 99.9% = 0.999

동기 직렬 호출 체인의 전체 가용성:
  2단계: 0.999 × 0.999 = 0.998 (99.8%) → 월 87분 다운
  3단계: 0.999³ = 0.997 (99.7%) → 월 131분 다운
  4단계: 0.999⁴ = 0.996 (99.6%) → 월 175분 다운
  5단계: 0.999⁵ = 0.995 (99.5%) → 월 219분 다운
  10단계: 0.999¹⁰ = 0.990 (99.0%) → 월 432분(7.2시간) 다운

→ 각 서비스는 99.9%를 보장해도 체인이 길수록 전체 가용성은 급락
```

**카스케이드 장애 시나리오:**

```
배송 서비스가 응답 지연 (3초) 상황:

클라이언트 → 주문 서비스 → 결제 서비스 → 재고 서비스 → 배송 서비스 (3초 대기)

결과:
1. 재고 서비스 스레드: 배송 서비스 응답 3초 대기 (스레드 점유)
2. 결제 서비스 스레드: 재고 서비스 응답 3초 대기 (스레드 점유)
3. 주문 서비스 스레드: 결제 서비스 응답 3초 대기 (스레드 점유)

동시 요청 100개 발생 시:
  각 서비스의 스레드 풀(20개) 소진
  → 새로운 요청은 스레드를 얻지 못해 즉시 에러
  → 배송 서비스 하나의 지연이 전체 시스템 마비
```

**비동기 이벤트로 개선:**

```
주문 서비스 → Kafka (OrderPlaced 이벤트 발행)
              ↓ [비동기]
결제 서비스 ← 이벤트 구독 (독립적 처리)
재고 서비스 ← 이벤트 구독 (독립적 처리)
배송 서비스 ← 이벤트 구독 (독립적 처리)

→ 각 서비스가 독립적으로 99.9% 가용성 유지
→ 배송 서비스 다운 시: 이벤트가 Kafka에 쌓이고 복구 후 처리
→ 카스케이드 장애 없음
```

---

### 안티패턴 3: 배포 순서 의존성

```
주문 서비스 v2.0 배포 계획:
  - OrderResponse에 deliveryFee 필드 추가

그런데:
  결제 서비스가 OrderResponse를 역직렬화 중
  → deliveryFee 필드가 없으면 결제 서비스 파싱 에러

배포 절차 강제됨:
  1. 결제 서비스에 deliveryFee 옵션 처리 코드 추가 먼저 배포
  2. 주문 서비스 v2.0 배포
  3. 결제 서비스에서 옵션 처리 코드 제거 배포

→ 3번의 조율된 배포, 롤백 시나리오 복잡
→ 진짜 독립 배포라면: Consumer-Driven Contract Testing으로 해결
```

## 💻 분산 모놀리스 진단 체크리스트

```
아래 중 하나라도 "예"라면 분산 모놀리스 징후:

□ 서비스 A 배포 시 서비스 B를 먼저 배포해야 한다
□ 서비스 A의 DB 스키마 변경 시 서비스 B 코드를 수정한다
□ 서비스 A가 서비스 B의 DB에 직접 쿼리한다
□ 서비스 B가 다운되면 서비스 A도 타임아웃으로 다운된다
□ 하나의 기능 변경이 3개 이상의 서비스를 동시에 수정하게 만든다
□ 서비스 간 API 계약이 문서화돼 있지 않다
□ 서비스 팀들이 매 배포마다 Slack으로 조율한다

점수: 3개 이상 → 분산 모놀리스
      5개 이상 → 모놀리스로 다시 합치는 것이 더 나을 수도
```

## 📊 패턴 비교 — 분산 모놀리스 vs 진짜 MSA

| 항목 | 분산 모놀리스 | 진짜 MSA |
|------|------------|---------|
| 데이터 | 공유 DB | 서비스별 독립 DB |
| 배포 | 순서 의존 | 언제든 단독 배포 |
| 스키마 변경 | 전체 팀 조율 필요 | 서비스 내부에서 자유롭게 |
| 장애 전파 | 한 서비스 다운 → 전체 영향 | Circuit Breaker로 격리 |
| 통신 | 동기 체인 | 이벤트 기반 + 필요 시 동기 |
| 단점 | 모놀리스+MSA 최악의 조합 | 운영 복잡도 |

## ⚖️ 트레이드오프

```
동기 호출 체인을 유지하는 이유:
  ✅ 즉시 응답 가능 (결과를 바로 알 수 있음)
  ✅ 구현 단순 (비동기 복잡도 없음)

동기 호출 체인을 제거해야 하는 이유:
  ❌ 체인 길이만큼 가용성 하락 (0.999^N)
  ❌ 카스케이드 장애 위험
  ❌ 호출 순서에 서비스가 결합

Shared Database를 유지하는 이유 (단기):
  ✅ 조인 쿼리 가능 → 데이터 조합 쉬움
  ✅ 트랜잭션 원자성 보장 쉬움

Shared Database를 제거해야 하는 이유 (장기):
  ❌ 스키마 변경이 전체 서비스에 영향
  ❌ DB 서버가 단일 실패점
  ❌ 독립 배포 불가
```

## 📌 핵심 정리

```
✅ 분산 모놀리스의 정의
   서비스를 물리적으로 분리했지만 데이터와 배포는 결합된 상태
   = 모놀리스의 단점 + MSA의 단점을 모두 가진 최악의 아키텍처

✅ 3가지 핵심 안티패턴
   1. Shared Database: 서비스가 같은 DB를 직접 접근
   2. 동기 호출 체인: 0.999^N — 체인 길이만큼 가용성 하락
   3. 배포 순서 의존성: A 배포 전에 B를 먼저 배포해야 함

✅ 가용성 공식
   직렬 동기 호출 N단계: 0.999^N
   5단계 체인 → 99.5% 가용성 → 월 219분 다운타임

✅ 탈출 방법
   1. DB 분리: 서비스별 독립 DB로 점진적 마이그레이션
   2. 이벤트화: 동기 호출 체인을 이벤트 기반으로 교체
   3. Circuit Breaker: 동기 호출 불가피 시 장애 격리
```

## 🤔 생각해볼 문제

**Q1.** 결제 서비스가 주문 서비스를 동기 호출하고, 재고 서비스도 주문 서비스를 동기 호출한다. 주문 서비스의 가용성이 99.9%일 때, 전체 흐름(주문→결제→재고)의 가용성은?

<details>
<summary>해설 보기</summary>

`0.999 × 0.999 × 0.999 = 0.997` → **99.7% 가용성**. 월 기준 약 131분의 다운타임을 의미한다. 각 서비스가 99.9%를 보장해도 직렬 체인은 그보다 낮은 가용성을 가진다. 비동기 이벤트로 전환하면 주문 서비스는 이벤트 발행만 하고, 결제·재고는 독립적으로 이벤트를 처리하므로 각자 99.9% 가용성을 유지할 수 있다.

</details>

**Q2.** 팀이 "서비스는 분리했지만 편의상 공유 DB를 잠깐만 쓰겠다"고 한다. 왜 이 결정이 위험한가?

<details>
<summary>해설 보기</summary>

"잠깐"이 영구화되는 이유가 있다. 공유 DB 위에서 코드가 쌓일수록 분리 비용이 기하급수적으로 증가한다. 6개월 후 수십 개의 JOIN 쿼리, 수백 개의 공유 테이블 참조가 생기면 분리가 사실상 불가능해진다. 처음부터 서비스별 DB를 분리하되, 초기에는 같은 DB 서버의 다른 스키마(schema)를 사용하는 방식으로 최소한의 격리를 유지하는 것이 훨씬 안전하다.

</details>

**Q3.** 배송 서비스가 다운됐을 때 주문 생성이 실패해서는 안 된다면, 어떻게 설계해야 하는가?

<details>
<summary>해설 보기</summary>

주문 생성과 배송 요청을 비동기로 분리한다. 주문 서비스는 주문을 DB에 저장하고 `OrderPlaced` 이벤트를 Kafka에 발행한다. 배송 서비스는 이 이벤트를 구독하여 배송을 요청한다. 배송 서비스가 다운된 동안에는 이벤트가 Kafka에 쌓이고, 복구되면 순서대로 처리한다. 주문 생성은 배송 서비스 상태와 무관하게 99.9% 가용성을 유지한다.

</details>
