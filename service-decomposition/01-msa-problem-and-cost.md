# 01. MSA가 해결하는 문제 — 모놀리스의 실제 한계

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 모놀리스는 왜 일정 규모를 넘으면 병목이 되는가?
- MSA는 모놀리스의 어떤 문제를 해결하고, 어떤 새로운 문제를 만드는가?
- 팀이 10명일 때와 100명일 때 아키텍처 선택이 왜 달라져야 하는가?
- "MSA로 전환하면 배포가 빨라진다"는 말이 항상 맞지 않는 이유는 무엇인가?
- 언제 모놀리스가 MSA보다 더 나은 선택인가?

---

## 🔍 왜 이 개념이 MSA에서 중요한가

MSA를 도입하는 이유를 명확히 이해하지 못하면, 아키텍처 전환 후에도 원래의 문제가 그대로 남는다. "MSA는 무조건 좋다"는 생각으로 전환을 시작하면, 운영 복잡도만 늘어난 분산 모놀리스를 만들게 된다. MSA가 무엇을 위해 설계됐는지 이해하는 것이 올바른 서비스 경계 설정의 출발점이다.

---

## 😱 흔한 실수 (Before — 원리를 모를 때의 접근)

```
상황: 배포 속도가 느리고 팀 간 충돌이 잦아 MSA 도입을 결정

원리를 모를 때의 접근:
  "MSA로 쪼개면 배포가 빨라지겠지"
  → 기능 단위로 서비스를 잘게 분리
  → 서비스끼리 직접 DB를 공유하는 구조 그대로 유지
  → 배포할 때마다 여전히 모든 팀에게 공지 필요
  → 분산된 모놀리스 완성 (운영 복잡도만 증가)

잘못된 판단의 근거:
  "서비스를 나누면 독립 배포가 된다"
  → 실제로는 공유 DB 스키마 변경 시 모든 서비스 영향
  "MSA는 빠르다"
  → 네트워크 홉이 추가되어 오히려 레이턴시 증가
  "서비스가 많을수록 MSA"
  → 서비스 수가 아니라 경계의 독립성이 핵심
```

## ✨ 올바른 접근 (After — 문제를 이해하고 결정)

MSA 도입 여부를 결정하기 전에 반드시 답해야 할 질문들이 있다.

```
✅ 모놀리스의 실제 문제를 먼저 진단한다

질문 1. 독립 배포가 안 되는가?
  → 팀 A의 기능을 배포하려면 팀 B, C의 승인이 필요한가?
  → 배포 브랜치 충돌이 주 1회 이상 발생하는가?

질문 2. 기술 스택이 종속됐는가?
  → ML 모델 서빙에 Python이 필요하지만 Spring 모놀리스에 끼워야 하는가?
  → 특정 팀만 Redis를 쓰고 싶지만 공용 의존성 충돌이 있는가?

질문 3. 팀 결합도가 높은가?
  → 팀 A가 코드를 수정하면 팀 B의 테스트가 깨지는가?
  → 한 팀의 배포 실패로 다른 팀의 릴리스가 취소되는가?

질문 4. 확장이 비효율적인가?
  → 주문 서비스만 트래픽이 몰리는데 전체 서버를 확장해야 하는가?
  → 특정 기능만 스케일아웃이 필요한 상황인가?

→ 위 질문에 "예"가 3개 이상이라면 MSA 전환을 고려할 시점
→ 아니라면 모놀리스 개선(모듈화)이 먼저
```

## 🔬 내부 동작 원리 — 모놀리스가 무너지는 구체적 시나리오

### 1. 독립 배포 불가 — 배포 결합도

```
모놀리스 배포 흐름:

팀 A (주문 기능) 개발 완료
팀 B (결제 기능) 개발 완료
팀 C (회원 기능) 아직 진행 중

                 하나의 배포 단위
┌─────────────────────────────────────┐
│  주문 코드 + 결제 코드 + 회원 코드  │
└─────────────────────────────────────┘
         ↑
팀 C가 준비될 때까지 팀 A, B도 배포 불가

결과:
- 배포 주기: 주 1회 → 팀 단위 딜레이 누적
- 긴급 핫픽스 시 전체 코드가 함께 배포됨
- 팀 A의 버그가 팀 B 기능을 롤백시킴
```

### 2. 기술 스택 종속 — 언어/프레임워크 종속

```
모놀리스: 전체가 Java 17 + Spring Boot 3.x

요구사항: 상품 추천 기능에 Python ML 모델 필요
         실시간 채팅에 Node.js WebSocket 서버 필요

선택지 1: Java에서 Python 프로세스를 subprocess로 호출
  → 운영 복잡도 증가, 에러 처리 어려움
  → ML 팀이 JVM 배포 파이프라인에 의존

선택지 2: 억지로 Java로 ML 구현
  → 성능 저하, 라이브러리 생태계 부족
  → 전문 ML 엔지니어가 Java를 학습해야 함

MSA라면:
  추천 서비스 → Python + FastAPI
  채팅 서비스 → Node.js
  주문 서비스 → Java + Spring
  각 서비스가 최적 기술로 독립 구현 가능
```

### 3. 확장 비효율 — 전체 스케일아웃

```
모놀리스 트래픽 분포:
  주문 서비스: CPU 80% (트래픽 집중)
  회원 서비스: CPU 5%
  상품 서비스: CPU 10%

스케일아웃 시:
  전체 모놀리스 인스턴스 3개 추가
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │주문+회원+상품+..│  │주문+회원+상품+..│  │주문+회원+상품+..│
  └──────────────────┘  └──────────────────┘  └──────────────────┘

  → 회원, 상품 서비스는 이미 여유롭지만 함께 복제됨
  → 메모리, CPU, 비용 낭비

MSA라면:
  주문 서비스만 3개 인스턴스로 확장
  회원, 상품 서비스는 그대로 유지
```

## 💻 MSA 전환이 가져오는 비용 — 공짜가 아닌 이유

MSA는 모놀리스의 문제를 해결하는 대신, 아래의 분산 시스템 문제를 전부 감수해야 한다.

### 네트워크 신뢰성 문제

```java
// 모놀리스: 직접 메서드 호출 — 실패 없음
OrderService orderService = new OrderService();
Order order = orderService.createOrder(request); // 항상 성공

// MSA: 네트워크 호출 — 언제든 실패 가능
@FeignClient(name = "order-service")
public interface OrderClient {
    @PostMapping("/orders")
    Order createOrder(OrderRequest request);
    // 타임아웃, 네트워크 단절, 서비스 다운 → 전부 처리해야 함
}
```

### 분산 트랜잭션 문제

```
모놀리스: 하나의 DB 트랜잭션으로 원자성 보장
  BEGIN TRANSACTION
    INSERT INTO orders ...
    INSERT INTO payments ...
    UPDATE inventory ...
  COMMIT — 셋 다 성공하거나 셋 다 실패

MSA: 서비스마다 DB가 분리됨
  Order Service → order_db에 INSERT
  Payment Service → payment_db에 INSERT
  Inventory Service → inventory_db에 UPDATE

  Payment는 성공, Inventory 업데이트 중 서비스 다운
  → 데이터 불일치 상태 발생
  → Saga 패턴으로 별도 처리 필요 (Chapter 4에서 다룸)
```

### 운영 복잡도 — 관찰해야 할 것이 N배

```
모놀리스:
  로그 파일 1개
  메트릭 대시보드 1개
  배포 파이프라인 1개

MSA (7개 서비스):
  로그 7개 → 중앙화 로깅 필요 (ELK Stack)
  메트릭 7개 → Prometheus + Grafana
  배포 파이프라인 7개 → CI/CD 자동화 필수
  서비스 간 트레이싱 → Zipkin / Jaeger
  서비스 디스커버리 → Eureka / Kubernetes
  API Gateway → Spring Cloud Gateway
  분산 설정 관리 → Spring Cloud Config
```

### 비용 비교

| 항목 | 모놀리스 | MSA |
|------|---------|-----|
| 서버 비용 | 낮음 (단일 인스턴스) | 높음 (서비스별 인스턴스) |
| 네트워크 비용 | 없음 (프로세스 내 호출) | 있음 (서비스 간 호출) |
| 개발 복잡도 | 낮음 | 높음 (분산 시스템 이해 필요) |
| 운영 복잡도 | 낮음 | 매우 높음 |
| 배포 독립성 | 불가 | 가능 |
| 장애 격리 | 불가 | 가능 (Circuit Breaker 필요) |
| 기술 자율성 | 제한 | 자유 |

## 📊 패턴 비교 — 모놀리스 vs 모듈러 모놀리스 vs MSA

```
1. 모놀리스 (Monolith)
   ┌─────────────────────────────────┐
   │ 주문 + 결제 + 회원 + 상품      │ ← 하나의 프로세스
   │ 하나의 DB, 하나의 배포 단위    │
   └─────────────────────────────────┘
   장점: 단순함, 낮은 운영 비용, 로컬 테스트 쉬움
   단점: 팀 규모 증가 시 배포 결합, 확장 비효율

2. 모듈러 모놀리스 (Modular Monolith)
   ┌─────────────────────────────────┐
   │  [주문 모듈] | [결제 모듈]     │ ← 하나의 프로세스
   │  [회원 모듈] | [상품 모듈]     │   모듈 간 인터페이스로 분리
   └─────────────────────────────────┘
   장점: 모놀리스의 단순함 + 코드 응집도
   단점: 여전히 단일 배포, 기술 스택 종속

3. MSA (Microservices)
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 주문 서비스│  │ 결제 서비스│  │ 회원 서비스│ ← 독립 프로세스
   │  order_db│  │ payment_db│  │  user_db │   독립 DB
   └──────────┘  └──────────┘  └──────────┘
   장점: 독립 배포, 독립 확장, 기술 자율성
   단점: 분산 시스템 복잡도 전부 감수
```

## ⚖️ 트레이드오프

### MSA가 해결하는 것

- **독립 배포**: 팀 A의 배포가 팀 B에게 영향 없음
- **독립 확장**: 트래픽이 몰리는 서비스만 스케일아웃
- **장애 격리**: 결제 서비스 다운이 상품 조회에 영향 없음 (Circuit Breaker 적용 시)
- **기술 자율성**: 서비스별 최적 언어/DB 선택 가능

### MSA가 만드는 새로운 문제

- **네트워크 신뢰성**: 모든 서비스 호출이 실패 가능 → 타임아웃, 재시도, Circuit Breaker 필요
- **분산 트랜잭션**: 여러 서비스에 걸친 데이터 일관성 → Saga 패턴 필요
- **분산 추적**: 요청이 어느 서비스에서 느린지 파악 → Zipkin/Jaeger 필요
- **운영 복잡도**: 관찰해야 할 서비스가 N개 → 전담 인프라 팀 또는 플랫폼 필요

### 팀 규모별 권장 선택

| 팀 규모 | 권장 아키텍처 | 이유 |
|---------|--------------|------|
| 1~5명 | 모놀리스 | 운영 복잡도 감당 불가 |
| 5~20명 | 모듈러 모놀리스 | 코드 분리는 필요하지만 MSA 오버헤드 불필요 |
| 20명 이상 | MSA 검토 | 팀 간 배포 결합이 실제 병목이 될 때 |
| 조직별 팀 | MSA | Conway's Law — 조직 구조가 아키텍처를 결정 |

## 📌 핵심 정리

```
✅ 모놀리스의 실제 문제
   1. 독립 배포 불가 — 하나의 팀이 전체 배포를 막음
   2. 기술 스택 종속 — 서비스 특성과 맞지 않는 기술을 강요당함
   3. 확장 비효율 — 트래픽이 집중된 서비스만 스케일아웃 불가
   4. 팀 결합도 — 코드 변경의 파급 범위가 팀 경계를 넘음

✅ MSA가 만드는 비용
   1. 네트워크 신뢰성 — 모든 서비스 호출이 실패 가능
   2. 분산 트랜잭션 — ACID 대신 Saga로 일관성 달성
   3. 운영 복잡도 — 로깅/모니터링/배포 파이프라인 N배
   4. 초기 투자 — 인프라 셋업, 학습 곡선

✅ 언제 MSA가 정답인가
   - 팀이 독립적으로 배포해야 할 때
   - 서비스마다 다른 기술 스택이 필요할 때
   - 특정 서비스만 집중적으로 확장해야 할 때
   - 조직 구조가 도메인별로 명확히 분리됐을 때

✅ 언제 모놀리스가 더 나은가
   - 팀이 10명 미만일 때
   - 도메인 경계가 아직 불명확할 때 (경계 설정 실수 비용이 큼)
   - 인프라 운영 전담 인력이 없을 때
```

## 🤔 생각해볼 문제

**Q1.** 팀이 5명인 스타트업이 MSA로 시작하기로 했다. 어떤 문제가 발생할 가능성이 높은가?

<details>
<summary>해설 보기</summary>

5명 팀에서 MSA를 도입하면 운영 복잡도가 개발 속도를 압도한다. 서비스 디스커버리(Eureka), API Gateway(Spring Cloud Gateway), 분산 추적(Zipkin), 중앙화 로깅(ELK)을 모두 셋업하고 유지하는 데 팀 리소스의 절반이 소진될 수 있다. 도메인 경계도 아직 불명확한 초기 단계에서 잘못 그은 서비스 경계는 나중에 재조정 비용이 크다. 이 시기에는 **모놀리스로 시작하되 모듈 경계를 명확히 유지**하다가, 배포 충돌과 팀 결합이 실제 병목이 될 때 MSA로 전환하는 Strangler Fig 전략이 적합하다.

</details>

**Q2.** "우리 팀은 MSA를 써서 배포가 빠르다"고 하는데, 실제로 확인해야 할 지표는 무엇인가?

<details>
<summary>해설 보기</summary>

배포 속도만으로는 MSA의 효과를 판단할 수 없다. 확인해야 할 지표는 세 가지다.

1. **배포 독립성**: 서비스 A를 배포할 때 서비스 B의 담당자에게 공지하거나 승인을 받는가? 그렇다면 독립 배포가 아니다.
2. **장애 격리**: 서비스 A가 다운됐을 때 서비스 B의 기능이 정상 동작하는가? Circuit Breaker 없이는 장애가 전파된다.
3. **데이터 독립성**: 서비스 A의 DB 스키마를 변경할 때 서비스 B의 코드를 함께 수정해야 하는가? 그렇다면 분산 모놀리스다.

</details>

**Q3.** 주문 서비스가 다운됐을 때 상품 목록 조회는 정상 동작해야 하는가? 이것이 MSA 설계에서 어떤 의미를 갖는가?

<details>
<summary>해설 보기</summary>

상품 목록 조회는 주문 서비스와 무관하므로 정상 동작해야 한다. 이것이 **장애 격리(Fault Isolation)** 원칙이다. 모놀리스에서는 주문 처리 로직의 버그가 상품 조회 스레드를 고갈시킬 수 있다. MSA에서는 서비스가 분리돼 있어 이론적으로 격리되지만, **Circuit Breaker와 Bulkhead 없이는** 주문 서비스로 가는 동기 호출이 타임아웃을 기다리며 스레드를 점유해 결국 상품 서비스까지 영향을 준다. 장애 격리는 서비스 분리만으로 달성되지 않고, 가용성 패턴(Chapter 5)과 함께 설계돼야 한다.

</details>
