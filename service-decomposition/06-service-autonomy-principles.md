# 06. 서비스 자율성 원칙 — 독립 배포·확장·기술·데이터

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 서비스 자율성의 4가지 기준(배포·확장·기술·데이터)이란 무엇인가?
- 각 기준을 위반했을 때 어떤 실제 문제가 발생하는가?
- 서비스가 "진짜 자율적"인지 판단하는 체크리스트는 무엇인가?
- 자율성을 높이면 반드시 생기는 비용은 무엇인가?
- 완전한 자율성을 추구하면 안 되는 경우는 언제인가?

---

## 🔍 왜 이 개념이 MSA에서 중요한가

서비스를 물리적으로 분리했다고 자율성이 생기지 않는다. 자율성은 4가지 차원에서 동시에 달성돼야 한다. 배포만 독립되고 데이터가 결합돼 있다면 반쪽 자율성이다. 이 장은 Chapter 1의 마무리로, 올바르게 분해된 서비스가 갖춰야 할 자율성의 전체 기준을 정의한다.

---

## 😱 흔한 실수 (Before — 자율성 착각)

```
"우리는 서비스가 분리됐으니 자율적입니다"

실제 점검:

배포 자율성 체크:
  주문 서비스를 배포할 때 결제 서비스 팀에 알려야 한다 → ❌
  배포 순서가 고정돼 있다 → ❌

확장 자율성 체크:
  주문 서비스만 스케일아웃하면 결제 서비스가 과부하된다 → ❌

기술 자율성 체크:
  모든 서비스가 같은 Spring Boot 버전을 써야 한다 → ❌

데이터 자율성 체크:
  주문 서비스 DB 스키마 변경 전에 결제팀 승인이 필요하다 → ❌
  다른 서비스가 내 DB에 직접 쿼리한다 → ❌

→ 4가지 중 하나라도 실패하면 완전한 자율성이 아님
```

---

## ✨ 올바른 접근 (After — 4가지 자율성 기준)

```
완전한 서비스 자율성 = 아래 4가지를 모두 만족

1. 독립 배포 자율성
   → 언제든 단독으로 배포할 수 있다
   → 다른 서비스 팀에 공지 없이도 안전하게 배포된다

2. 독립 확장 자율성
   → 트래픽이 몰리는 이 서비스만 스케일아웃할 수 있다
   → 다른 서비스에 영향 없이 독립적으로 확장된다

3. 기술 스택 자율성
   → 이 서비스에 최적인 언어/프레임워크/DB를 자유롭게 선택한다
   → 다른 서비스의 기술 변경이 이 서비스에 영향 없다

4. 데이터 자율성
   → 이 서비스만 자신의 DB를 읽고 쓴다
   → DB 스키마 변경이 다른 서비스 코드를 수정하게 만들지 않는다
```

---

## 🔬 내부 동작 원리 — 4가지 자율성 상세 분석

### 1. 독립 배포 자율성

**API 버저닝 — Breaking Change 없이 배포:**

```
v1 (기존):
  GET /orders/{id}
  Response: { "orderId": "...", "status": "..." }

v2 (신규 필드 추가):
  Response: { "orderId": "...", "status": "...", "deliveryFee": 3000 }
  → 필드 추가는 하위 호환 → 기존 클라이언트는 deliveryFee 무시
  → 주문 서비스 v2 배포 시 결제 서비스 코드 변경 불필요 ✅

Breaking Change 시:
  GET /api/v1/orders/{id}  → 기존 응답 형식 유지
  GET /api/v2/orders/{id}  → 새 응답 형식
  → v1, v2 병행 운영 → 클라이언트 준비되면 v2로 전환
```

**Consumer-Driven Contract Testing:**

```java
@Pact(consumer = "payment-service")
public RequestResponsePact createPact(PactDslWithProvider builder) {
    return builder
        .given("order exists")
        .uponReceiving("get order by id")
        .path("/orders/order-123")
        .method("GET")
        .willRespondWith()
        .status(200)
        .body(new PactDslJsonBody()
            .stringValue("orderId", "order-123")
            .stringValue("status", "PENDING")
            .numberValue("totalAmount", 50000))
        .toPact();
    // 계약이 깨지면 주문 서비스 배포 전에 CI에서 감지
}
```

### 2. 독립 확장 자율성

```yaml
# Kubernetes HPA — 주문 서비스만 자동 확장
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    name: order-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
# → CPU가 낮아지면 Pod 자동 축소
# → 결제 서비스는 그대로 유지
```

**독립 확장을 위한 Stateless 설계:**

```
❌ 잘못된 예: 세션을 서버 메모리에 저장
  → 인스턴스 2로 요청이 가면 세션 없음 → 로그인 풀림

✅ 올바른 예:
  세션 → Redis에 저장 (공유 세션 스토어)
  JWT → 클라이언트에 저장 (서버 무상태)
```

### 3. 기술 스택 자율성

```
자율성 허용 범위 (외부에 노출되지 않는 내부 기술):
  ✅ 언어: Java, Python, Go, Node.js
  ✅ 프레임워크: Spring, FastAPI, Express
  ✅ DB: MySQL, MongoDB, Redis, Elasticsearch
  ✅ 내부 라이브러리 버전

제한 필요 (서비스 간 계약에 영향):
  ⚠️ API 직렬화 포맷: JSON vs Protobuf (합의 필요)
  ⚠️ 인증 방식: JWT 검증 방식 통일 필요

현실적 권장:
  "기술 자율성을 허용하되, 기본 스택을 정하고
   이유가 명확할 때만 다른 기술 선택"
```

### 4. 데이터 자율성

```
데이터 자율성의 3가지 규칙:

규칙 1: 서비스 DB는 해당 서비스만 접근
  → 다른 서비스는 API를 통해서만 데이터 요청

규칙 2: 필요한 데이터는 복제하여 소유
  → 주문 서비스가 상품명이 필요하면:
     OrderPlaced 시 productName 스냅샷을 order_items에 저장

규칙 3: 스키마 변경은 하위 호환 방식으로
  → 컬럼 추가: 언제든 가능
  → 컬럼 삭제: Deprecated 후 모든 사용처 제거 완료 후 삭제
  → 컬럼 타입 변경: 새 컬럼 추가 → 이관 → 구 컬럼 삭제
```

## 💻 자율성 점검 체크리스트

```
[ 독립 배포 자율성 ]
□ 주문 서비스를 배포할 때 다른 팀에 사전 공지가 필요한가?     → NO여야 통과
□ 배포 후 결제 서비스가 영향받는 경우가 있는가?              → NO여야 통과
□ Consumer-Driven Contract Test가 CI에 포함됐는가?           → YES여야 통과

[ 독립 확장 자율성 ]
□ 주문 서비스 Pod를 5개로 늘릴 때 결제 서비스도 늘려야 하는가? → NO여야 통과
□ 주문 서비스가 Stateless하게 설계됐는가?                   → YES여야 통과

[ 기술 스택 자율성 ]
□ Spring Boot 버전 업그레이드 시 다른 팀 동의 필요?          → NO여야 통과

[ 데이터 자율성 ]
□ 다른 서비스가 order_db에 직접 쿼리하는가?                  → NO여야 통과
□ 스키마 변경 시 다른 팀 승인이 필요한가?                    → NO여야 통과

점수: 전부 통과 → 완전한 자율성
      일부 실패 → 분산 모놀리스 위험
```

## ⚖️ 트레이드오프

```
자율성을 높일수록 생기는 비용:

배포 자율성 비용:
  - Consumer-Driven Contract Test 도구 셋업 및 유지
  - API 버저닝 관리, 구버전 유지 기간 이중 구현

확장 자율성 비용:
  - Stateless 설계를 위한 세션 외부화 (Redis)
  - 스케일아웃 시 DB 연결 수 급증 (Connection Pool 관리)

기술 자율성 비용:
  - 폴리글랏 환경 운영 복잡도
  - 공통 관심사(로깅, 모니터링) 각 언어별 구현

데이터 자율성 비용:
  - 조인 없는 데이터 조합 → API Composition 또는 CQRS
  - 데이터 복제로 인한 일관성 관리 (Eventual Consistency)

→ 자율성은 공짜가 아니다
```

## 📌 핵심 정리

```
✅ 서비스 자율성 4가지 기준
   1. 독립 배포: 언제든 단독 배포 가능, 다른 팀 공지 불필요
   2. 독립 확장: 이 서비스만 스케일아웃 가능
   3. 기술 자율성: 내부 기술 스택 자유 선택
   4. 데이터 자율성: 서비스가 자신의 DB를 단독 소유

✅ Chapter 1 마무리
   서비스 자율성은 올바른 분해(Ch1)의 결과물
   자율성을 유지하면서 데이터 일관성은 → Chapter 3, 4에서 해결
   자율성을 유지하면서 장애를 격리하는 방법 → Chapter 5에서 해결
```

## 🤔 생각해볼 문제

**Q1.** 결제 서비스가 주문 서비스의 DB에 직접 접근하는 현재 코드가 있다. 제거하면서 기능을 유지하는 가장 안전한 방법은?

<details>
<summary>해설 보기</summary>

3단계로 진행한다. 1단계: 주문 서비스에 결제 서비스가 필요로 하는 데이터를 반환하는 API 엔드포인트를 추가한다. 2단계: 결제 서비스의 직접 DB 접근 코드를 주문 서비스 API 호출로 교체한다. Feature Flag로 두 경로를 병행 운영한다. 3단계: 직접 DB 접근 코드를 제거하고 주문 서비스 DB의 결제 서비스 계정 권한을 회수한다.

</details>

**Q2.** 서비스 자율성과 표준화(공통 라이브러리, 공통 설정)는 어떻게 균형을 잡아야 하는가?

<details>
<summary>해설 보기</summary>

"외부 계약은 표준화, 내부 구현은 자율"이 원칙이다. 표준화가 필요한 것: JWT 검증, 로깅 포맷, OpenTelemetry. 이것들은 플랫폼 팀이 제공하는 공통 인프라다. 자율성이 보장돼야 하는 것: 내부 데이터 모델, 비즈니스 로직, DB 스키마. 공통 라이브러리를 강제 도입하면 버전 업그레이드가 전체 서비스 동시 배포를 유발한다. "플랫폼으로 표준화하되, 코드 레벨 공유는 최소화"가 실용적 균형점이다.

</details>

**Q3.** "MSA를 도입했다"고 말할 수 있는 최소 조건은 무엇인가?

<details>
<summary>해설 보기</summary>

최소 조건 세 가지다. 첫째, 데이터 자율성: 각 서비스가 독립 DB를 소유하고 다른 서비스가 직접 접근하지 않는다. 둘째, 독립 배포: 서비스 하나를 배포할 때 다른 팀의 동의나 동시 배포가 필요 없다. 셋째, 장애 격리: 하나의 서비스 장애가 다른 서비스에 카스케이드되지 않는다. 나머지(기술 자율성, 확장 자율성)는 이 세 가지 달성 후 점진적으로 개선해도 된다.

</details>
